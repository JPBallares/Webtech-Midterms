W W W (World Wide Web)

Overview :

WWW stands for World Wide Web is a collection of web resources and applications that identified by Uniform Resource Locators (URLs), interlinked by hypertext links that can be access via the internet. World Wide Web was invented by Tim Berners-Lee and co-inventor Robert Cailliau in 1989 at CERN in Geneva. The main goal of Tim Berners-Lee for inventing WWW is for the researcher around the world for them share information.

W W W (World Wide Web)

Internet :

- Global network of networks.
- A collection of hardware and software (e.g. Hosts, Media, Protocols).
- Uses internet protocol suite (TCP/IP) to link devices worldwide.

H T T P (Hypertext Transfer Protocol)

Overview :

HTTP stands for Hypertext Transfer Protocol that is used by the web. It defines how the information is transmitted and formatted. HTTP is an application layer communications protocol used to access resources (Hypertext/Hypermedia) on the World Wide Web. Develop by Tim Berners-Lee in 1989 at CERN. Port 80 is the default port.

H T T P (Hypertext Transfer Protocol)

HTTP Message :

HTTP Message is use to communicate between the client and the server. The client sends a request message and the server will respond sending a response message corresponding to the request of the client. The Request messages is automatically created when accessing resources using browsers, but it can be create manually. The Response message is also automatically created but it will depend on the server's configuration.

H T T P (Hypertext Transfer Protocol)

Request Message :

Request Message is a message sent by the client to the server in order to access resources. It is compose of Request Line, Message Headers, Empty Line (CRLF) , and Pay Load or Body.

Request Message Parts :

Request Line

- Compose of request method, request target, and protocol version.

Request Method

- It informs the server what the client wants to do with the resources.

Request Target

- The URI or Uniform Resource Identifier that identifies the resources that the client wants to access is located ( asterisk "*" can be used to refer to the server itself ).

Protocol Version

- Provides the version of the HTTP that the client uses, so that the server knows what HTTP version to return to the client. (E.g HTTP/1.1).

Request Message Headers

- This allows client to provide more information to the server so that the server will be able to know what the client is capable of doing or what the client can handle. This information is placed in Request Header Fields.

Empty Line

- Indicates that the end of the Header Fields has been reached.

Message Body or Payload

- An optional part of a request message that contains a resource which can be sent to the server (using PUT method) or updated by the server.

H T T P (Hypertext Transfer Protocol)

Response Message :

Response Message is a message created by the server to answer the request of the client. It is compose Status Line, Message Headers, Empty Line (CRLF), and the Message Body or Payload.

Response Message Parts :

Status Line

- Compose of Protocol Version, Status Code, and Status Description.

Protocol Version

- Provides the HTTP version of the Response Message.

Status Code

- Code compose of 3 digit that describes what happen to the request of the client, if it is successfulor failure. Some examples are 200, 404, 304.

Status Description

- A brief description of the Status Code (human-readable). Examples: Ok, Not Found, Not modified. Example of Status Line: HTTP/1.1 200 OK, HTTP/1.1 404 Not Found, HTTP/1.1 304 Not Modified

H T T P (Hypertext Transfer Protocol)

GET :

- Retrieve full resources from the server, can also be used to submit information but it is not advisable because the information can be seen on the URI.

H T T P (Hypertext Transfer Protocol)

HEAD :

- Retrieve meta-data information of the resources, without the message body or the content of the resources.

H T T P (Hypertext Transfer Protocol)

PUT :

- Submit new resources on the server, or edit an existing resource in the server, the resources to be sent or modified is placed in the message body.

H T T P (Hypertext Transfer Protocol)

DELETE :

- Remove the resources that from the server that is identified by the URI specified by the client. This will fail if the server doesn't have the resource identified by the client.

H T T P (Hypertext Transfer Protocol)

OPTIONS :

- Shows the possible method that can be applied to the resources identified by the URI. This can also be applied to the server itself ( by using * as the target ) this will show the allowed method by the server.

H T T P (Hypertext Transfer Protocol)

TRACE :

- This will allow to track down where did the request message pass before getting into the server, this can also help client to see the exact message that arrives at the server.

H T T P (Hypertext Transfer Protocol)

CONNECT :

- Use proxy to establish tunnel to the remote server. It usually is being used in secured connection.

H T T P (Hypertext Transfer Protocol)

Safe Methods :

- This methods are only used as retrieval of resources ( read-only ), therefor this request message will not affect the server or the other client accessing the same resources and can be considered safe. Methods that are unsafe should be disabled, except for some special cases, by the server to avoid client from doing unnecessary action and the server can work properly.

H T T P (Hypertext Transfer Protocol)

Idempotent Method :

- This methods are considered idempotent because the expected result of this request methods to the server will be the same even if it is requested single time or it is requested multiple times(except caching and errors). This can be identified easily because when failure occurs in the communication the client can just resend the request to the server and expect that the result will be done.

H T T P (Hypertext Transfer Protocol)

Cacheable Methods :

- It is possible to store the current request and reuse it when other client request with the same request message. In some cases POST request method will not be cache depending on the configuration of the server.

H T T P (Hypertext Transfer Protocol)

General Headers :

- Is being used by the both client and server in sending messages ( request and response messages ) and has no relation to the body is is being sent. Some examples are Date, Cache Control, Pragma, Connection, Transfer Encoding, Upgrade, and Warning.

H T T P (Hypertext Transfer Protocol)

Request Headers :

- Provide information about the client or the resource that the client want to access. Some examples are Accept, Accept-*, If-*, Cookie, User-Agent, Host and Referer.

H T T P (Hypertext Transfer Protocol)

Response Headers :

- Provide information about the response being sent or the information about the server. Some examples are Age, Location, and Server.

H T T P (Hypertext Transfer Protocol)

Entity Headers :

- Is used to provide information about the message body or payload, like content length and content type. Some examples are Content-Type, Content-Length, Content-Language and Content-Encoding.

H T T P (Hypertext Transfer Protocol)

Content Negotiation :

- allows HTTP clients to negotiate for the resources that it prefer using and the server can provide for the available

Accept

- Provide the type of the resources to be accessed by the client.

Accept-Charset

- Provide the information on character sets that can be interpreted by the client.

Accept-Encoding

- Provide the type of encoding that can be interpreted by the client, this usually refers to the compression that the server can handle.

Accept-Language

- Provide the expected language to be accessed by the client. The client should provide a sequence of language sin case the first language is not available.

H T T P (Hypertext Transfer Protocol)

Connection Management :

Connection

- Manage the network connection to open or close the current communication after it is finished.

Keep-Alive

- Set how long a connection can stay open after the communication is finished.

H T T P (Hypertext Transfer Protocol)

Range Requests :

- Allow the client to access part of the resources.

Range

- Included in the client request, to specify a partial request of the resource.

Accept-Range

- Indicates if the server allows partial request on the resources.

If-Range

- Provides condition for the range request, range request will only be successful if the condition is met.

Content-Range

- Specify part of the resources to be accessed.

H T T P (Hypertext Transfer Protocol)

Authentication and Authorization :

- Allows the server to restrict resources that can be accessed by the client, and allows the client to provide their credentials to access the file.

WWW-Authenticate

- Describes how the client can access the resources.

Authorization

- Provides the type of authentication (Basic Authentication, Digest Authentication) and the client credentials. The credentials is usually encrypted.

Proxy-Authenticate

- Describes how the client can be access the resources in a Proxy Server.

Proxy-Authorization

- Provides the type of authentication ( Basic Authentication, Digest Authentication ) and the client credentials in a proxy server.

H T T P (Hypertext Transfer Protocol)

Conditionals :

- Specify conditions that must be met in order for the client to access the resources or in order for the server to send the response.

Last-Modified

- Provides the date of modification of the resources, it can be used to validate the client's cache and the server's resource.

Etag

- This uniquely identify the resource's version, this will be used as a condition when checking for client's cache and the server's resource.

If-Match

- A condition that checks if the Etags of the client's caches matches the Etags of the resources being accessed before doing the operation.

If-None-Match

- A condition that checks if the Etags of the client's caches didn't match any of the Etags in the servers resources. Usually used to update the cache of the client or to prevent duplication of resources.

If-Modified-Since

- A condition that checks if the resource is modified since a given date. This can help to determine if the cache is out of date.

If-Unmodified-Since

-A condition that checks if the resources is not modified since a given date.

H T T P (Hypertext Transfer Protocol)

Status Codes :

- Values of the http requests.

H T T P (Hypertext Transfer Protocol)

1XX Informational

- Provisional response.
- No required headers.

100 Continue

- Client should continue its requests.
- Client should should send the remaining requests.

101 Switching Protocols

- The requester asked the server to switch protocols and acknowleding that it will do so.

102 Processing ( WebDAV; RFC 2518 )

- The server has received the processing request but no response is available yet.

H T T P (Hypertext Transfer Protocol)

2xx Success :

- Server understands the request of clients, accepts and processed it successfully.

200 OK

- The request was successful.

201 Created

- Created a new resource and fulfilled.

202 Accepted

- Process has been accepted but not completed.

203 Non-Authoritative Information ( since HTTP/1.1 )

- Successfully processed the request but the information came from another source.

204 No Content

- Successfully processed the request but no contents returned.

205 Reset Content

- Successfully processed the request but no contents returned.
- This requires the requester to reset the document view.

206 Partial Content ( RFC 7233 )

- Delivers only part of a resource due to a range header sent by the client.

207 Multi-Status ( WebDAV; RFC 4918 )

- It can contain number of separate response codes, depending on how many sub-requests were made.

208 Already reported ( WebDAV; RFC 5842 )

- DAV agreement have formerly been counted and listed in an earlier reply to the request and aren't being incorporated any more.

226 IM Used ( RFC 3229 )

- The server fulfilled a request for the source, however, the response is a demonstration of the resul of at least one or more instance-manipulation assigned to the current instance.

H T T P (Hypertext Transfer Protocol)

3XX Redirection :

- Clients must take additional action to complete the request.

300 Multiple Choices

- Indicates multiple options for the resource that the client may follow.

301 Moved Permanently

- This and all future requests should be directed to the selected URL.

302 Found

- The requested resource resides temporarily under a different URI.

303 See Other ( since HTTP/1.1 )

- The response to the request can be found under a different URI and SHOULD be retrieved using a GET method on that resource. This method exists primarily to allow the output of a POST-activated script to redirect the user agent to a selected resource. The new URI is not a substitute reference for the originally requested resource.

304 Not Modified ( RFC 7232 )

- Indicates the resource has not been modified since last requested.

305 Use Proxy ( since HTTP/1.1 )

- The requested resource is only available through a proxy.

306 Switch Proxy

- This status response is no longer in use.

307 Temporary Redirect ( since HTTP/1.1 )

- The requested resource resides temporarily under a different URI.

308 Permanent Redirect ( RFC 7538 )

- Indicates that the current request and all forthcoming requests should be repeated using another URI.

H T T P (Hypertext Transfer Protocol)

4xx Client Error :

- Intended for the cases which the client seems to have erred.

400 Bad Request

- The request cannot be fulfilled due to bad syntax.

401 Unauthorized ( RFC 7235 )

- The request requires user authentication.

402 Payment Required

- Reserved for future use.

403 Forbidden

- Request was invalid.

404 Not Found

- The requested resource could not be found but may be available again in the future.

405 Method Not Allowed

- A request was made of a resource using a request method not supported by that resource.

406 Not Acceptable

- The requested resource is only capable of generating content not acceptable according to the Accept headers sent in the request.

407 Proxy Authentication Required ( RFC 7235 )

- The client must first authenticate itself with the proxy.

408 Request Time-out

- This type of response means that the server has timed out while waiting for the request.

409 Conflict

- Indicates that the request could not be processed because of conflict in the request, such as an edit conflict.

410 Gone

- The request resource has been permanently removed.

411 Length Required

- Refuses the request of invalid Content-Length.

412 Precondition Failed ( RFC 7232 )

- Indicates that access to the target resource has been denied.

413 Payload Too Large ( RFC 7231 )

- The request entity is large and the server is not willing to process.

414 URI Too Long ( RFC 7231 )

- The request URI is too large and the server is not willing to interpret.

415 Unsupported Media Type

- The entity is not valid format for the requested method and not supported by the request resource.

416 Range Not Satisfiable ( RFC 7233 )

- A server cannot serve the requested ranges.
- Occurs in most webtrafic.

417 Expectation Failed

- The request's Expect header could not be met.

418 I'm a teapot ( RFC 2324 )

- The server refuses to brew coffee because it is a teapot. This was originated in Hyper Text Coffee Pot Control Protocol.

421 Misdirected Request ( RFC 7540 )

- The request was directed at a server that is not able to produce a response.

422 Unprocessable Entity ( WebDAV; RFC 4918 )

- It accepts the request but unable to process the contained instruction.

423 Locked ( WebDAV; RFC 4918 )

- The destination that is being accessed is locked.

424 Failed Dependency ( WebDAV; RFC 4918 )

- The request failed because on another request that failed.

426 Upgrade Required

- The server will only accept the request after upgrading the needed protocol.

428 Precondition Required ( RFC 6585 )

- The server requires the request to be conditional.

429 Too Many Requests ( RFC 6585 )

- The user give too many request at the given amount time.

431 Request Header Fields Too Large ( RFC 6585 )

- The server is unwilling to process the request because some or all the headers are too large.

451 Unavailable For Legal Reasons

- The server is denying access to the resource as a consequence of a legal reasons.

H T T P (Hypertext Transfer Protocol)

5xx Server Error

- The server is aware that it is on error or is incapable of performing the request.
- It explains also if it is temporary or permanent condition.

500 Internal Server Error

- The server encountered an unexpected condition that prevented it from fulfilling the request.

501 Not Implemented

- The server do not recognised or lacks the ability to perform the request.

502 Bad Gateway

- The server was acting as a gateway or proxy and received an invalid response from the upstream server.

503 Service Unavailable

- The server is temporarily unavailable.

504 Gateway Time-out

- The server was acting as a gateway or proxy and did not receive a timely response from the upstream server.

505 HTTP Version Not Supported

-The version of the HTTP is not supported by the server.

506 Variant Also Negotiates ( RFC 2295 )

- The chosen variant resource is configured to engage in transparent content negotiation itself.

507 Insufficient Storage ( WebDAV; RFC 4918 )

- The server is unable to store the representation needed to complete the request.

508 Loop Detected ( WebDAV; RFC 5842 )

- The server detected an infinite loop while processing the request.

510 Not Extended ( RFC 2774 )

- The extensions to the request are required for the server to process it.

511 Network Authentication Required ( RFC 6585 )

- The client needs to authenticate to gain network access.
